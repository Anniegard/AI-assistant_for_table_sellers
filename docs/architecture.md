# Архитектура

Проект построен как слоистое приложение с разделением ответственности.

## Слои

1. Telegram UI: принимает сообщения пользователя и отображает ответы.
2. Assistant dialogue layer: persona, intent routing, business-aware ответ и CTA.
3. Product catalog: хранит товары и выполняет структурную фильтрацию.
4. Knowledge base: загружает и ищет статьи и FAQ в markdown.
5. AI layer: формирует объяснения и ответы на основе контекста и правил.
6. Lead collection: нормализует и сохраняет лиды.
7. Notification/export layer: интерфейс для будущей отправки менеджеру и Google Sheets.
8. Future web widget: следующий канал после Telegram MVP.

## Ключевой архитектурный принцип

- Товары: только через структурную фильтрацию по параметрам.
- FAQ и статьи: отдельный knowledge search, позже расширяемый до RAG.
- LLM не может придумывать товарные позиции: она только объясняет варианты, уже найденные каталогом.

Нельзя заменять фильтрацию товарного каталога векторной базой, потому что параметры выбора должны быть точными и воспроизводимыми.

## UX flow (web + Telegram)

- В обоих каналах используется единый пошаговый сбор параметров:
  `scenario -> height -> budget -> monitors -> pc_on_desk -> size`.
- На каждом шаге работает free-text parser с учетом `active_step`, чтобы корректно различать, например, одиночное число как рост или бюджет.
- После первой рекомендации сессия не сбрасывается: поддерживаются пост-интенты
  (`подороже`, `есть дешевле`, смена сценария/размера/бюджета, `почему советуете`).

## Recommender rules

- Сценарии пользователя маппятся на наборы каталожных тегов и влияют на scoring, а не только на жесткий фильтр по одному `use_case`.
- Бюджет учитывается как диапазон (`budget_min`/`budget_max`).
- При нехватке результатов в строгом бюджете включается fallback до `budget_max * 1.15` c пометкой компромисса в ответе.
